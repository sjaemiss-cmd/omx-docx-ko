비개발자를 위한 쉬운 풀이
  • 이 문서의 핵심은 “AI에게 그냥 한 번에 시키는 것”을 넘어, 계획·역할 배정·상태 저장·검증을 갖춘 작업 운영 방식으로 Codex CLI를 확장하는 것입니다.
  • 명령 예시는 그대로 복사해 쓸 수 있도록 보존했으며, 주변 설명은 “왜 필요한지”를 이해하기 쉽게 덧붙였습니다.
이 섹션에서 나오는 용어 풀이
  • Oh My Codex / OMX — Codex CLI를 더 체계적으로 쓰게 해 주는 보조 도구입니다. 여러 AI 작업자를 불러 역할을 나누고, 작업 상태를 저장하고, 검증까지 이어 가도록 돕습니다.
  • Codex CLI — 터미널에서 OpenAI Codex를 사용하는 명령줄 프로그램입니다. 마우스로 누르는 앱이 아니라 글자로 명령을 입력해 쓰는 도구라고 보면 됩니다.
  • CLI — Command Line Interface의 줄임말입니다. 버튼 대신 터미널에 글자를 입력해서 프로그램을 실행하는 방식입니다.
  • Hermes MCP — OMX 상태, 산출물, 로그 일부, 후속 작업 지시를 안전한 범위 안에서 다른 도구가 볼 수 있게 해 주는 연결 통로입니다.
  • MCP — Model Context Protocol의 줄임말입니다. AI와 외부 도구가 정보를 주고받는 약속된 연결 방식입니다.
  • 워크플로우 — 여러 단계를 정해진 순서대로 진행하는 일 처리 흐름입니다.
  • 플러그인 — 기본 프로그램에 기능을 덧붙이는 확장 꾸러미입니다.
  • $ultraqa / UltraQA — 일부러 까다로운 상황을 만들어 결과가 버티는지 확인하는 강한 품질검사 흐름입니다.
  • e2e — end-to-end의 줄임말입니다. 사용자가 처음부터 끝까지 실제로 쓰는 전체 흐름을 검사한다는 뜻입니다.
  • 프로젝트 메모리 — 프로젝트의 기술 스택, 규칙, 중요한 결정을 저장해 다음 작업에서도 참고하게 하는 기억 저장소입니다.
  • Ultragoal — 큰 목표를 여러 단계로 나누고, 끝났다는 증거가 충분한지 확인하게 하는 목표 관리 흐름입니다.

비개발자를 위한 쉬운 설명과 용어 풀이

읽는 방법

아래 원래 문서 내용은 삭제하지 않고 그대로 보존했습니다. 대신 각 섹션 끝에 쉬운 풀이를 추가하고, 낯선 기술 용어는 이 용어집에서 풀어 설명합니다.

명령어, 파일명, 환경 변수처럼 그대로 입력해야 하는 값은 영어 또는 코드 형태를 유지했습니다. 괄호 안 설명은 “무슨 뜻인지” 이해하기 위한 보조 설명입니다.

이 문서에서 “작업자”, “에이전트”, “모드”, “스킬”은 모두 AI가 일을 나누어 처리하도록 돕는 운영 단위입니다.

Oh My Codex / OMX

Codex CLI를 더 체계적으로 쓰게 해 주는 보조 도구입니다. 여러 AI 작업자를 불러 역할을 나누고, 작업 상태를 저장하고, 검증까지 이어 가도록 돕습니다.

Codex CLI

터미널에서 OpenAI Codex를 사용하는 명령줄 프로그램입니다. 마우스로 누르는 앱이 아니라 글자로 명령을 입력해 쓰는 도구라고 보면 됩니다.

CLI

Command Line Interface의 줄임말입니다. 버튼 대신 터미널에 글자를 입력해서 프로그램을 실행하는 방식입니다.

터미널

명령어를 글자로 입력하는 검은 화면 또는 콘솔 창입니다.

명령어

컴퓨터에게 시키는 짧은 문장입니다. 예: npm install -g oh-my-codex 는 설치하라는 뜻입니다.

npm

Node.js 생태계의 프로그램 설치 도구입니다. 스마트폰 앱스토어처럼 자바스크립트 도구를 내려받는 창구입니다.

설치

내 컴퓨터에서 프로그램을 사용할 수 있도록 파일을 내려받고 등록하는 과정입니다.

setup

OMX가 필요한 설정 파일, 역할 프롬프트, 스킬, 연결 도구를 준비하는 초기 설정 단계입니다.

doctor

설치가 제대로 되었는지 건강검진처럼 점검하는 명령입니다.

resume

이전에 하던 Codex 작업 세션을 다시 이어 여는 기능입니다.

에이전트

특정 역할을 맡은 AI 작업자입니다. 예를 들어 설계 담당, 구현 담당, 검토 담당처럼 역할이 나뉩니다.

프롬프트

AI에게 역할, 규칙, 해야 할 일을 알려 주는 지시문입니다.

스킬

자주 쓰는 작업 절차를 묶어 둔 실행 방법입니다. 예: 계획 세우기, 코드 리뷰, 알림 설정.

워크플로우

여러 단계를 정해진 순서대로 진행하는 일 처리 흐름입니다.

파이프라인

계획 → 요구사항 정리 → 실행 → 검증 → 수정처럼 작업이 흐르는 단계 묶음입니다.

오케스트레이션

여러 작업자와 도구가 서로 엉키지 않게 순서와 역할을 조율하는 일입니다. 지휘자가 오케스트라를 지휘하는 것과 비슷합니다.

Conductor / 지휘자

직접 모든 코드를 쓰기보다, 적절한 AI 작업자에게 맡기고 결과를 검토하는 사람 또는 리더 역할을 뜻합니다.

위임

작업을 직접 하지 않고 알맞은 역할의 에이전트에게 맡기는 것입니다.

검증

결과가 정말 맞는지 테스트, 빌드, 실행 확인 등으로 증거를 모으는 단계입니다.

릴리스

새 버전을 공개하는 일입니다.

패치 릴리스

큰 기능 추가보다는 버그 수정이나 안정화 중심의 작은 버전 공개입니다.

마이너 릴리스

기존 사용법을 크게 깨지 않으면서 기능을 추가하는 버전 공개입니다.

버전

프로그램의 공개 번호입니다. v0.17.0처럼 적으며, 어떤 기능과 수정이 포함됐는지 구분합니다.

Hermes MCP

OMX 상태, 산출물, 로그 일부, 후속 작업 지시를 안전한 범위 안에서 다른 도구가 볼 수 있게 해 주는 연결 통로입니다.

MCP

Model Context Protocol의 줄임말입니다. AI와 외부 도구가 정보를 주고받는 약속된 연결 방식입니다.

브리지

서로 다른 도구나 시스템을 이어 주는 다리 역할의 기능입니다.

아티팩트

작업 결과로 남는 파일, 보고서, 계획서, 로그 같은 산출물입니다.

로그

프로그램이 어떤 일을 했는지 시간순으로 남긴 기록입니다.

로그 tail

로그 전체가 아니라 가장 최근 부분만 잘라 보는 것입니다.

디스패치

후속 작업이나 메시지를 알맞은 대상에게 보내는 동작입니다.

플러그인

기본 프로그램에 기능을 덧붙이는 확장 꾸러미입니다.

마켓플레이스

플러그인이나 기능을 찾아 설치할 수 있는 목록 또는 저장소입니다.

캐시

다시 쓰기 위해 임시로 저장해 둔 데이터입니다. 매번 새로 받지 않아 속도가 빨라집니다.

메타데이터

파일이나 기능 자체가 아니라 그것을 설명하는 부가 정보입니다. 예: 이름, 버전, 범위.

DESIGN.md

$design 워크플로가 남기는 설계 문서 파일입니다. 제품 방향, 화면 결정, 중요한 이유를 오래 보관합니다.

$design

제품이나 화면 설계 결정을 문서로 정리하는 OMX 스킬입니다.

$ultraqa / UltraQA

일부러 까다로운 상황을 만들어 결과가 버티는지 확인하는 강한 품질검사 흐름입니다.

e2e

end-to-end의 줄임말입니다. 사용자가 처음부터 끝까지 실제로 쓰는 전체 흐름을 검사한다는 뜻입니다.

적대적 테스트

일부러 실패하기 쉬운 상황, 잘못된 입력, 공격적인 요청을 넣어 문제가 없는지 보는 테스트입니다.

프롬프트 인젝션

사용자 입력 안에 AI의 원래 지시를 무시하게 만들려는 문구를 넣는 공격 방식입니다.

오래된 상태

현재 상황과 맞지 않는 예전 작업 상태나 기록입니다. 잘못 사용하면 엉뚱한 결론이 날 수 있습니다.

런타임

프로그램이 실제로 실행되고 있는 시간과 그 실행 환경을 말합니다.

후크

특정 사건이 생겼을 때 자동으로 실행되는 연결 지점입니다. 예: 시작할 때, 멈출 때 알림 보내기.

Windows 네이티브 후크

Windows 환경에서 별도 우회 없이 직접 동작하도록 만든 자동 실행 연결 기능입니다.

tmux

터미널 하나 안에서 여러 작업 창을 나누고 유지하는 도구입니다. 여러 AI 작업자를 동시에 띄울 때 사용됩니다.

셸

터미널 명령을 해석해 실행하는 프로그램입니다. Bash, PowerShell 등이 있습니다.

프로젝트 메모리

프로젝트의 기술 스택, 규칙, 중요한 결정을 저장해 다음 작업에서도 참고하게 하는 기억 저장소입니다.

Ultragoal

큰 목표를 여러 단계로 나누고, 끝났다는 증거가 충분한지 확인하게 하는 목표 관리 흐름입니다.

worktree

하나의 Git 저장소에서 작업 공간을 여러 개로 나눈 것입니다. 여러 사람이 동시에 고쳐도 서로 파일을 덮어쓰지 않게 돕습니다.

Git

파일 변경 이력을 기록하고 되돌릴 수 있게 하는 버전 관리 도구입니다.

저장소 / repository

프로젝트 파일과 변경 이력이 함께 들어 있는 작업 폴더입니다.

브랜치

Git에서 독립적으로 작업하기 위해 갈라 만든 작업 줄기입니다.

커밋

Git에 “여기까지의 변경을 하나의 기록으로 저장한다”라고 남기는 단위입니다.

머지 / merge

서로 다른 브랜치나 작업 결과를 하나로 합치는 일입니다.

체리픽 / cherry-pick

다른 브랜치의 특정 커밋 하나만 골라 가져오는 일입니다.

리베이스 / rebase

내 작업의 출발점을 최신 기준 위로 다시 얹어 정리하는 Git 작업입니다.

충돌 / conflict

두 작업자가 같은 부분을 다르게 고쳐 Git이 자동으로 합치지 못하는 상태입니다.

팀 모드

여러 AI 작업자 창을 띄워 한 목표를 나누어 처리하게 하는 OMX 실행 방식입니다.

작업자 / worker

팀 모드에서 실제로 분석, 수정, 검증 같은 일을 맡는 개별 AI 실행 단위입니다.

리더

팀 전체 방향을 정하고 작업자를 배정하며 최종 결과를 검토하는 중심 역할입니다.

Codex / Claude / Gemini

서로 다른 AI 도구 또는 AI 실행 환경입니다. OMX는 이들을 섞어 팀처럼 쓸 수 있습니다.

혼합 공급자 팀

Codex, Claude, Gemini 같은 여러 AI 도구를 한 팀 안에서 함께 쓰는 방식입니다.

모델

AI의 두뇌에 해당하는 실행 엔진입니다. 빠른 모델과 깊이 생각하는 모델이 구분됩니다.

모델 라우팅

쉬운 일은 빠른 모델에, 어려운 일은 더 깊게 생각하는 모델에 보내는 자동 배정입니다.

Spark

빠른 탐색과 간단한 정리에 쓰는 가벼운 모델 등급입니다.

Standard

일반 구현, 테스트, 리팩터링에 쓰는 기본 모델 등급입니다.

xhigh reasoning

AI가 더 많은 사고 시간을 쓰도록 하는 높은 추론 설정입니다.

yolo 모드

승인 절차를 줄이고 빠르게 실행하는 모드입니다. 편하지만 위험할 수 있습니다.

madmax

승인과 샌드박스 제한을 크게 우회하는 위험한 실행 옵션입니다. 믿을 수 있는 환경에서만 써야 합니다.

샌드박스

프로그램이 마음대로 시스템을 건드리지 못하게 제한하는 안전 울타리입니다.

읽기 전용

파일을 바꾸지 않고 보기만 하는 상태입니다.

허용 목록 / allowlist

실행해도 된다고 미리 허락한 명령이나 경로 목록입니다.

경로 이탈 / path escape

정해진 프로젝트 폴더 밖의 파일을 몰래 보거나 건드리려는 상황입니다.

경로 탐색 / path traversal

../ 같은 경로 조작으로 허용되지 않은 파일에 접근하려는 보안 문제입니다.

설정 파일

프로그램이 어떻게 동작할지 적어 두는 파일입니다. config.toml이나 .omx-config.json이 예입니다.

config.toml

Codex와 OMX 설정을 적는 TOML 형식의 설정 파일입니다.

환경 변수

터미널이나 운영체제에 임시로 저장해 프로그램이 읽게 하는 설정값입니다.

JSON

데이터를 중괄호와 키-값 형태로 적는 형식입니다. 사람이 읽고 프로그램도 읽기 쉽습니다.

TOML

설정 파일에 자주 쓰는 읽기 쉬운 텍스트 형식입니다.

웹훅 / webhook

어떤 사건이 생기면 지정된 인터넷 주소로 자동 메시지를 보내는 방식입니다.

봇 토큰

Telegram이나 Discord 봇을 식별하고 권한을 주는 비밀 키입니다. 공개하면 안 됩니다.

채널 ID / chat ID

알림을 보낼 대화방이나 채널을 가리키는 번호입니다.

헤더 / header

웹 요청에 함께 붙는 부가 정보입니다. 인증 키를 넣는 데 자주 쓰입니다.

인증 / auth

누가 요청했는지 확인하고 허락된 사람인지 검사하는 절차입니다.

POST / PUT

웹 서버로 데이터를 보낼 때 쓰는 요청 방식입니다. POST는 새로 보내기, PUT은 지정 위치에 넣거나 갱신하는 의미로 자주 씁니다.

알림 이벤트

세션 시작, 멈춤, 종료, 사용자 질문처럼 알림을 보낼 계기가 되는 사건입니다.

reply injection / 응답 주입

Telegram, Discord, Slack 답장을 Codex 세션 입력으로 넣어 원격으로 대화에 참여하는 기능입니다.

LSP

Language Server Protocol의 줄임말입니다. 코드 편집기가 정의 찾기, 참조 찾기, 오류 표시를 하도록 돕는 표준입니다.

AST Grep

코드를 단순 문자열이 아니라 문법 구조로 찾아 바꾸는 도구입니다.

Python REPL

Python 명령을 한 줄씩 실행해 결과를 바로 보는 실험용 실행 창입니다.

리팩터링

겉으로 보이는 기능은 유지하면서 내부 코드를 더 읽기 쉽고 안전하게 정리하는 일입니다.

디버깅

버그의 원인을 찾아 고치는 과정입니다.

PRD

Product Requirements Document의 줄임말입니다. 만들 기능의 목적, 요구사항, 성공 기준을 적은 문서입니다.

API

프로그램끼리 서로 기능을 호출하기 위해 정한 약속입니다.

공개 API 파손

다른 사람이 쓰던 호출 방식이 바뀌어 기존 사용자가 고장나는 상황입니다.

동적 스케일링

일하는 도중 작업자 수를 늘리거나 줄이는 기능입니다.

scale_up / scale_down

작업자를 늘리는 명령과 줄이는 명령입니다.

잠금 / lock

동시에 같은 작업을 건드리지 못하게 막는 안전 장치입니다.

메일박스

팀 작업자와 리더가 메시지를 주고받는 내부 전달함입니다.

상태 루트

팀 상태 파일을 어디에 저장하고 읽을지 정하는 기준 폴더입니다.

HUD

화면 위에 현재 상태를 요약해 보여 주는 표시 영역입니다.

OpenClaw

알림과 후속 작업 트리거를 더 유연하게 다루기 위한 외부 연동 게이트웨이입니다.

#작업트리 오케스트레이션(v0.10.0의 기본값)

v0.10.0의 Game-changer

팀 작업자는 이제 격리된 git worktrees에서 실행됩니다. 기본적으로. 는 --worktree 플래그는 더 이상 사용되지 않으며 no-op로 처리됩니다. worktrees는 항상 켜져 있습니다.

작동 방식

  1. 깨끗한 리더 작업 공간 필요 — 리더는 먼저 헌신적인/stashed 작업 공간을 가지고 있어야 합니다. omx team 시작합니다.
  2. 자동 worktree 프로비저닝 — 각 작업자는 분리된 worktree를 얻습니다. .omx/team/<name>/worktrees/worker-N.
  3. 근로자는 worktree에 전념합니다. — 격리된 분기는 작업자 간의 쓰기 충돌이 없음을 의미합니다.
  4. 리더는 점진적으로 병합됩니다. — 런타임은 작업자 커밋을 리더 브랜치에 지속적으로 통합합니다.
  5. 종료 정리 — 프로비저닝된 worktrees는 팀 종료 시 롤백되고 분기가 삭제됩니다.

증분 병합 전략

통합 엔진은 작업자별로 최상의 전략을 자동으로 선택합니다.

  • 병합(--no-ff -X theirs) — 작업자가 리더보다 확실히 앞서 있을 때. 빠르고 충돌-free.
  • 체리-pick — 역사가 갈라졌을 때. 개별 작업자 커밋을 리더 브랜치로 선택합니다.
  • Cross-worker 리베이스 — 작업자 간의 순차적 종속성을 위해. 한 작업자의 분기를 다른 작업자의 통합 상태로 리베이스합니다.

충돌이 조기에 감지되어 기록됩니다. .omx/state/team/<team>/integration-report.md per-file 디테일로.

작업자 커밋 프로토콜

작업자는 완료를 보고하기 전에 다음을 커밋해야 합니다. git add -A && git commit -m "task: <subject>". 작업자가 잊어버린 경우를 대비해 런타임 auto-commits를 사용하지만, 더 깔끔한 기록을 위해 명시적인 커밋을 선호합니다.

BASH
# Worktrees are automatic — no flags needed
omx team 3:executor "implement feature X"

# Check worker worktree status
omx team status <team-name>
# Output includes: worktree_path, worktree_branch, worktree_detached per worker

# Mixed providers — all get worktrees
OMX_TEAM_WORKER_CLI_MAP=codex,claude,gemini omx team 3:executor "full-stack work"

Cross-Worktree 상태 확인

  • 팀 상태 루트는 리더 cwd(<leader-cwd>/.omx/state)를 통해 모든 worktrees에서 공유됩니다. OMX_TEAM_STATE_ROOT.
  • 파견, 작업 청구, 사서함 배달 및 수명 주기 업데이트는 모든 작업자 worktrees에서 일관성을 유지합니다.
  • 작업자 신원 파일 추적 worktree_path, worktree_branch, worktree_detached, 그리고 worktree_created 완전한 관찰이 가능하도록 합니다.

#Gemini-Powered 팀

  • 전체 지원 Gemini CLI OMX 팀 모드의 작업자.
  • 복잡한 아키텍처 계획 및 코드베이스-wide 리팩터링을 위해 Gemini의 대규모 컨텍스트 창과 추론 기능을 활용하세요.
  • 혼합 공급자 팀(Codex + Claude + Gemini)은 최고의 "최적 조합" 오케스트레이션을 가능하게 합니다.
  • 다음으로 구성 OMX_TEAM_WORKER_CLI_MAP=codex,claude,gemini.

런타임 예

전문적인 추론 능력을 활용하려면 Gemini로 혼합 팀을 운영하세요.

BASH
# 1. Start a mixed provider team
export OMX_TEAM_WORKER_CLI_MAP=codex,claude,gemini
omx team 3:executor "full-stack implementation"

# 2. Force all workers to Gemini
export OMX_TEAM_WORKER_CLI=gemini
omx team 2:architect "large-scale architectural review"

# 3. Pass explicit Gemini models
export OMX_TEAM_WORKER_LAUNCH_ARGS="--model gemini-2.0-flash-exp"
omx team 1:executor "experimental feature"

Gemini 작업자 행동 참고 사항:

  • 프롬프트-interactive 시작: Gemini 작업자는 다음을 사용하여 초기 프롬프트로 자동으로 시드됩니다. --approval-mode yolo -i "..." 안정적인 tmux Bring-up. 보장
  • 모델 필터링: Non-Gemini 기본 모델(예: Spark)은 Gemini 작업자를 시작할 때 자동으로 필터링되어 공급자/model 불일치를 방지합니다.
  • 상황적 이점: 심층적인 코드베이스 컨텍스트 또는 long-form 아키텍처 합성이 필요한 작업에는 Gemini 워커를 사용하세요.
비개발자를 위한 쉬운 풀이
  • 기능 가이드는 운영자가 자주 쓰는 큰 기능을 빠르게 이해하도록 모아 둔 안내입니다.
  • 동적 스케일링은 작업자 수 조절, RALPLAN-DR은 여러 관점의 계획 검토, OpenClaw는 알림/후속 작업 연결, worktree는 안전한 병렬 작업 공간을 뜻합니다.
이 섹션에서 나오는 용어 풀이
  • 동적 스케일링 — 일하는 도중 작업자 수를 늘리거나 줄이는 기능입니다.
  • scale_up / scale_down — 작업자를 늘리는 명령과 줄이는 명령입니다.
  • 잠금 / lock — 동시에 같은 작업을 건드리지 못하게 막는 안전 장치입니다.
  • OpenClaw — 알림과 후속 작업 트리거를 더 유연하게 다루기 위한 외부 연동 게이트웨이입니다.
  • 알림 이벤트 — 세션 시작, 멈춤, 종료, 사용자 질문처럼 알림을 보낼 계기가 되는 사건입니다.
  • worktree — 하나의 Git 저장소에서 작업 공간을 여러 개로 나눈 것입니다. 여러 사람이 동시에 고쳐도 서로 파일을 덮어쓰지 않게 돕습니다.
  • Git — 파일 변경 이력을 기록하고 되돌릴 수 있게 하는 버전 관리 도구입니다.
  • 커밋 — Git에 “여기까지의 변경을 하나의 기록으로 저장한다”라고 남기는 단위입니다.
  • 머지 / merge — 서로 다른 브랜치나 작업 결과를 하나로 합치는 일입니다.
  • 체리픽 / cherry-pick — 다른 브랜치의 특정 커밋 하나만 골라 가져오는 일입니다.
  • 리베이스 / rebase — 내 작업의 출발점을 최신 기준 위로 다시 얹어 정리하는 Git 작업입니다.
  • 충돌 / conflict — 두 작업자가 같은 부분을 다르게 고쳐 Git이 자동으로 합치지 못하는 상태입니다.

#왜 OMX인가?

Codex CLI는 직접적인 작업에 강력합니다. OMX는 대규모 작업을 위한 구조를 추가합니다.

  • 분해 및 단계적 실행: 다음을 통해 관리됨 team-plan → team-prd → team-exec → team-verify → team-fix 파이프라인.
  • 영구 모드 수명 주기 상태: 저장 위치 .omx/state/ 재개 가능한 실행을 위해.
  • 메모리 + 메모장 표면: long-running 세션을 위한 전용 작업 공간입니다.
  • 운영 통제: Fine-grained는 실행, 확인, 취소를 제어합니다.

#지휘자 철학

OMX의 핵심 원리는 다음과 같습니다. 당신은 공연자가 아니라 지휘자입니다.

황금률

NEVER는 코드를 직접 변경합니다. ALWAYS는 전문 에이전트에게 위임합니다. 귀하의 역할은 안내하고, 검토하고, 조정하는 것입니다.

상담원에게는 특별한 역할이 있습니다. 는 건축가 큰 그림을 본다, 집행자 코드를 작성하고, 검증인 그것이 효과가 있음을 증명합니다. 이 구분을 존중하면 더 높은 품질의 출력이 보장됩니다.

비개발자를 위한 쉬운 풀이
  • OMX의 철학은 사용자가 모든 일을 직접 하기보다, 목적을 정하고 AI 작업자들이 역할별로 일하도록 지휘하는 것입니다.
  • 비개발자 관점에서는 “한 명의 만능 직원”이 아니라 “기획자, 개발자, 검수자가 있는 작은 팀”을 운영한다고 이해하면 됩니다.
이 섹션에서 나오는 용어 풀이
  • Conductor / 지휘자 — 직접 모든 코드를 쓰기보다, 적절한 AI 작업자에게 맡기고 결과를 검토하는 사람 또는 리더 역할을 뜻합니다.
  • 위임 — 작업을 직접 하지 않고 알맞은 역할의 에이전트에게 맡기는 것입니다.
  • 에이전트 — 특정 역할을 맡은 AI 작업자입니다. 예를 들어 설계 담당, 구현 담당, 검토 담당처럼 역할이 나뉩니다.
  • 검증 — 결과가 정말 맞는지 테스트, 빌드, 실행 확인 등으로 증거를 모으는 단계입니다.
  • 오케스트레이션 — 여러 작업자와 도구가 서로 엉키지 않게 순서와 역할을 조율하는 일입니다. 지휘자가 오케스트라를 지휘하는 것과 비슷합니다.

  • PRD — Product Requirements Document의 줄임말입니다. 만들 기능의 목적, 요구사항, 성공 기준을 적은 문서입니다.
  • 작업자 / worker — 팀 모드에서 실제로 분석, 수정, 검증 같은 일을 맡는 개별 AI 실행 단위입니다.
  • 리더 — 팀 전체 방향을 정하고 작업자를 배정하며 최종 결과를 검토하는 중심 역할입니다.
  • 검증 — 결과가 정말 맞는지 테스트, 빌드, 실행 확인 등으로 증거를 모으는 단계입니다.

OMX는 작업을 가장 적절한 모델 계층에 지능적으로 라우팅하여 비용과 기능의 균형을 맞춥니다.

복잡성모델사용 사례

  • 간단한 검색은 빠른 모델, 복잡한 설계나 원인 분석은 더 깊이 생각하는 모델을 쓰면 시간과 비용을 균형 있게 관리할 수 있습니다.

  • 모델 라우팅 — 쉬운 일은 빠른 모델에, 어려운 일은 더 깊게 생각하는 모델에 보내는 자동 배정입니다.
  • Spark — 빠른 탐색과 간단한 정리에 쓰는 가벼운 모델 등급입니다.
  • Standard — 일반 구현, 테스트, 리팩터링에 쓰는 기본 모델 등급입니다.
  • xhigh reasoning — AI가 더 많은 사고 시간을 쓰도록 하는 높은 추론 설정입니다.

#팀 구성

v0.10.0

아래의 모든 팀 구성은 다음에서 실행됩니다. 격리된 기본적으로 git worktree. 각 작업자는 자체 worktree를 자동으로 가져옵니다. 병합 충돌이 없고 증분 통합이 가능하며 완전한 종료 롤백이 가능합니다.

#MCP 도구

상태 및 런타임

  • 상태 도구: state_read, state_write, state_clear 실행 모드 상태를 관리합니다.
  • 메모장: notepad_read, notepad_write_priority, notepad_write_working, notepad_write_manual.

코드 인텔리전스

LSP 도구: lsp_hover, lsp_goto_definition, lsp_find_references, lsp_diagnostics, lsp_rename.

AST 그렙: ast_grep_search, ast_grep_replace 구조적 리팩토링을 위해.

유틸리티

파이썬 REPL: 데이터 분석을 위한 지속적인 환경.

MCP WorkingDirectory 정책(선택적 강화)

기본적으로 MCP 상태/memory/trace 도구는 호출자를 수락합니다-provided workingDirectory. 이를 제한하려면 루트 허용 목록을 설정하세요.

BASH
export OMX_MCP_WORKDIR_ROOTS="/path/to/project:/path/to/another-root"

설정되면, workingDirectory 이러한 루트 외부의 값은 보안 도구에 의해 거부됩니다.

비개발자를 위한 쉬운 풀이
  • MCP 도구는 AI가 프로젝트 상태, 코드 정보, 메모리 같은 외부 자료를 안전하게 읽고 쓰도록 해 주는 연결 도구입니다.
  • 상태 도구는 현재 작업 단계 저장, LSP는 코드 위치 찾기, AST Grep은 코드 구조 검색, Python REPL은 작은 실험 실행에 해당합니다.
이 섹션에서 나오는 용어 풀이
  • MCP — Model Context Protocol의 줄임말입니다. AI와 외부 도구가 정보를 주고받는 약속된 연결 방식입니다.
  • 상태 루트 — 팀 상태 파일을 어디에 저장하고 읽을지 정하는 기준 폴더입니다.
  • 프로젝트 메모리 — 프로젝트의 기술 스택, 규칙, 중요한 결정을 저장해 다음 작업에서도 참고하게 하는 기억 저장소입니다.
  • LSP — Language Server Protocol의 줄임말입니다. 코드 편집기가 정의 찾기, 참조 찾기, 오류 표시를 하도록 돕는 표준입니다.
  • AST Grep — 코드를 단순 문자열이 아니라 문법 구조로 찾아 바꾸는 도구입니다.
  • Python REPL — Python 명령을 한 줄씩 실행해 결과를 바로 보는 실험용 실행 창입니다.
  • 경로 이탈 / path escape — 정해진 프로젝트 폴더 밖의 파일을 몰래 보거나 건드리려는 상황입니다.
  • 허용 목록 / allowlist — 실행해도 된다고 미리 허락한 명령이나 경로 목록입니다.

비개발자를 위한 쉬운 풀이
  • CLI 참고는 터미널에서 입력할 수 있는 명령어 사전입니다.
  • 각 명령은 “무엇을 하는지”를 설명하며, 위험한 옵션은 왜 조심해야 하는지도 함께 풀어 두었습니다.
이 섹션에서 나오는 용어 풀이
  • CLI — Command Line Interface의 줄임말입니다. 버튼 대신 터미널에 글자를 입력해서 프로그램을 실행하는 방식입니다.
  • 명령어 — 컴퓨터에게 시키는 짧은 문장입니다. 예: npm install -g oh-my-codex 는 설치하라는 뜻입니다.
  • 터미널 — 명령어를 글자로 입력하는 검은 화면 또는 콘솔 창입니다.
  • resume — 이전에 하던 Codex 작업 세션을 다시 이어 여는 기능입니다.
  • 읽기 전용 — 파일을 바꾸지 않고 보기만 하는 상태입니다.
  • 허용 목록 / allowlist — 실행해도 된다고 미리 허락한 명령이나 경로 목록입니다.
  • yolo 모드 — 승인 절차를 줄이고 빠르게 실행하는 모드입니다. 편하지만 위험할 수 있습니다.
  • madmax — 승인과 샌드박스 제한을 크게 우회하는 위험한 실행 옵션입니다. 믿을 수 있는 환경에서만 써야 합니다.
  • 샌드박스 — 프로그램이 마음대로 시스템을 건드리지 못하게 제한하는 안전 울타리입니다.
  • Spark — 빠른 탐색과 간단한 정리에 쓰는 가벼운 모델 등급입니다.
  • worktree — 하나의 Git 저장소에서 작업 공간을 여러 개로 나눈 것입니다. 여러 사람이 동시에 고쳐도 서로 파일을 덮어쓰지 않게 돕습니다.

#알림

OMX 세션 상태가 변경되거나 입력이 필요할 때 경고를 받습니다.

개요

  • OMX는 Codex 세션이 시작, 중지, 유휴 상태가 되거나 입력을 요청할 때 경고를 보낼 수 있습니다.
  • Discord(웹훅), Discord(봇 API), Telegram, Slack 및 일반 웹훅의 5가지 플랫폼이 지원됩니다.
  • 다음에서 알림을 구성할 수 있습니다. config.toml 또는 환경 변수를 사용합니다.
팁

환경 변수로 시작한 다음 실행하십시오. omx setup 구성 파일을 작성합니다.

빠른 설정

가장 빠른 설정은 환경 변수를 내보낸 다음 설정을 실행하는 것입니다.

BASH

플랫폼

OMX는 다양한 플랫폼을 지원하여 어디서든 정보를 얻을 수 있습니다.

플랫폼 주요 구성 필드 세부정보
Telegram botToken, chatId 지원 parseMode (마크다운 또는 HTML). 응답 주입을 위해 승인된 채팅 ID에 응답합니다.
Discord(웹훅) webhookUrl 간단한 one-way 경고. 맞춤형 지원 username 그리고 mention 대상(예: <@123456>).
Discord (봇 API) botToken, channelId two-way 통신에 권장됩니다. 다음을 갖춘 Discord 봇이 필요합니다. Send Messages 권한.
Slack webhookUrl 맞춤형 지원 channel, username, 그리고 mention 대상(예: <!here>).
일반 웹훅 url POST 또는 PUT JSON 페이로드는 모든 엔드포인트에 적용됩니다. 맞춤형 지원 headers 인증을 위해.

OpenClaw 구성

OpenClaw는 프로덕션-grade 알림 오케스트레이션에 권장되는 경로입니다. 이를 통해 에이전트 회전 및 복잡한 팔로우-up 동작을 트리거할 수 있습니다.

JSON(오픈클로 프로파일)
{
  "notifications": {
    "verbosity": "verbose",
    "openclaw": {
      "enabled": true,
      "gateways": {
        "local": {
          "type": "http",
          "url": "http://127.0.0.1:18789/hooks/agent",
          "headers": { "Authorization": "Bearer ${HOOKS_TOKEN}" }
        }
      },
      "hooks": {
        "session-start": {
          "enabled": true,
          "gateway": "local",
          "instruction": "[session-start|exec] project={{projectName}} session={{sessionId}}"
        },
        "ask-user-question": {
          "enabled": true,
          "gateway": "local",
          "instruction": "[ask-user-question|exec] session={{sessionId}} question={{question}}"
        }
      }
    }
  }
}

자세한 내용은 다음을 참조하세요. 전체 OpenClaw 통합 가이드.

이벤트

이벤트 화재가 발생하면
session-startCodex 세션이 시작되면 발생합니다.
session-stop세션이 중지되면 발생합니다(모드 완료 또는 사용자 중단).
session-end세션이 종료되면 발생합니다.
session-idle세션이 유휴 상태(입력 대기 중)일 때 발생합니다.
ask-user-question에이전트가 사용자에게 질문을 하면 실행됩니다.

비개발자를 위한 쉬운 풀이
  • 알림은 작업 시작, 멈춤, 질문 필요 같은 상황을 Telegram, Discord, Slack 등으로 알려 주는 기능입니다.
  • 토큰, 웹훅 URL, 채널 ID는 비밀값이거나 대상 주소이므로 공개 저장소에 올리지 않는 것이 안전합니다.
이 섹션에서 나오는 용어 풀이
  • 알림 이벤트 — 세션 시작, 멈춤, 종료, 사용자 질문처럼 알림을 보낼 계기가 되는 사건입니다.
  • 웹훅 / webhook — 어떤 사건이 생기면 지정된 인터넷 주소로 자동 메시지를 보내는 방식입니다.
  • 봇 토큰 — Telegram이나 Discord 봇을 식별하고 권한을 주는 비밀 키입니다. 공개하면 안 됩니다.
  • 채널 ID / chat ID — 알림을 보낼 대화방이나 채널을 가리키는 번호입니다.
  • 헤더 / header — 웹 요청에 함께 붙는 부가 정보입니다. 인증 키를 넣는 데 자주 쓰입니다.
  • 인증 / auth — 누가 요청했는지 확인하고 허락된 사람인지 검사하는 절차입니다.
  • POST / PUT — 웹 서버로 데이터를 보낼 때 쓰는 요청 방식입니다. POST는 새로 보내기, PUT은 지정 위치에 넣거나 갱신하는 의미로 자주 씁니다.
  • 환경 변수 — 터미널이나 운영체제에 임시로 저장해 프로그램이 읽게 하는 설정값입니다.
  • reply injection / 응답 주입 — Telegram, Discord, Slack 답장을 Codex 세션 입력으로 넣어 원격으로 대화에 참여하는 기능입니다.
  • OpenClaw — 알림과 후속 작업 트리거를 더 유연하게 다루기 위한 외부 연동 게이트웨이입니다.

#권장되는 워크플로우

일반적인 작업을 위한 Battle-tested 패턴입니다. 맞는 것을 고르세요.

PRD의 전체-Auto

먼저 계획이 필요한 큰 기능의 경우.

$ralplan → $team → $ralph
  1. $ralplan는 기획자+건축가+비평가의 합의를 통해 요구사항을 수집하고 계획을 수립합니다.
  2. $team는 병렬로 빌드할 병렬 tmux 작업자를 생성합니다.
  3. $ralph는 모든 것이 완료되었음을 확인할 때까지 지속됩니다.

아니요-Brainer

작업이 명확하고 완료되기를 원할 때.

$autopilot → $ultrawork → $ralph
  1. $autopilot는 바퀴를 잡고 자율적으로 작동합니다.
  2. $ultrawork는 속도를 위해 하위 작업을 병렬화합니다.
  3. $ralph는 확인될 때까지 지속됩니다("돌은 절대 멈추지 않습니다").

수정/디버깅

버그 및 오류의 경우.

$plan → $ralph → $ultraqa
  1. $plan는 문제를 조사하고 수정 전략을 만듭니다.
  2. $ralph는 수정 사항을 구현하고 완료될 때까지 계속 진행됩니다.
  3. $ultraqa는 모든 테스트가 통과할 때까지 테스트 -> 확인 -> 수정을 순환합니다.

병렬 문제 처리

여러 문제나 티켓을 동시에 처리합니다.

omx team (건축가) → omx team (노동자) → $ralplan → $ralph + $ultrawork → $ultraqa
  1. 건축가 작업자를 시작하여 모든 문제를 분석하고 하나의 완전한 계획 초안을 작성하십시오.

  4. 모든 테스트가 통과할 때까지 $ralph, $ultrawork 및 $ultraqa로 완료합니다.

이 네 가지 패턴은 대부분의 실제-world 작업을 다룹니다. 특정 작업을 위한 다른 기술이 존재하지만 거의 필요하지 않습니다. day-to-day.

비개발자를 위한 쉬운 풀이
  • 권장 워크플로우는 자주 쓰는 작업 조합입니다. 큰 기능은 계획 후 팀 실행, 명확한 작업은 자동 실행, 버그는 계획·수정·검증 순서가 좋습니다.
  • 화살표는 실제 일이 흘러가는 순서를 뜻하며, 앞 단계 결과가 뒤 단계의 입력이 됩니다.
이 섹션에서 나오는 용어 풀이
  • 워크플로우 — 여러 단계를 정해진 순서대로 진행하는 일 처리 흐름입니다.
  • PRD — Product Requirements Document의 줄임말입니다. 만들 기능의 목적, 요구사항, 성공 기준을 적은 문서입니다.
  • 팀 모드 — 여러 AI 작업자 창을 띄워 한 목표를 나누어 처리하게 하는 OMX 실행 방식입니다.
  • $ultraqa / UltraQA — 일부러 까다로운 상황을 만들어 결과가 버티는지 확인하는 강한 품질검사 흐름입니다.
  • 검증 — 결과가 정말 맞는지 테스트, 빌드, 실행 확인 등으로 증거를 모으는 단계입니다.

#고급 오케스트레이션

대규모 엔지니어링의 경우 OMX는 복잡한 팀 구성, 혼합 CLI 작업자 및 git worktree 격리.를 지원합니다.

혼합 CLI 팀

단일 팀 내에서 다양한 AI 제공업체(Codex, Claude 및 Gemini)를 혼합하여 고유한 강점을 활용할 수 있습니다.

BASH
# Mix Codex, Gemini, and Claude workers in one team
export OMX_TEAM_WORKER_CLI_MAP=codex,gemini,claude
omx team 3:executor "refactor the shared runtime docs"

팀 CLI API

모든 변형 및 조정은 안정적인 JSON-based 상태 전환을 위해 CLI 상호 운용성 API를 사용해야 합니다. 이는 자동화된 도구 및 복잡한 작업자 조정에 매우 중요합니다.

BASH
# Create a task programmatically
omx team api create-task --input '{"team_name":"my-team","subject":"Fix bug","description":"..."}' --json

# Claim a task with versioned safety
omx team api claim-task --input '{"team_name":"my-team","task_id":"1","worker":"worker-1"}' --json

# Transition terminal task status after claiming
omx team api transition-task-status --input '{"team_name":"my-team","task_id":"1","from":"in_progress","to":"completed","claim_token":""}' --json

작업 트리 격리

전용 전용 git worktree에서 팀을 실행하면 병렬 작업자가 서로의 파일을 훼손하는 것을 방지합니다. OMX는 이러한 worktree 전체에서 상태 확인을 자동으로 처리합니다.

BASH
# Launch in a new worktree
omx team 4:executor "feature work" --worktree=feature-branch
아키텍처 통찰력

OMX는 모든 작업에 대해 내구성 있는 클레임-safe 수명 주기를 유지합니다. 작업자가 실패하거나 세션이 중단된 경우 상태는 .omx/state/team/, 안전한 이력서를 허용합니다. omx team resume <team-name>.

비개발자를 위한 쉬운 풀이
  • 고급 오케스트레이션은 여러 AI 도구, 여러 작업 공간, 자동 상태 전환을 섞어 큰 작업을 안정적으로 처리하는 방법입니다.
  • 비개발자는 “여러 전문가를 동시에 움직이되, 작업 기록과 합치는 절차를 시스템이 관리한다”라고 이해하면 됩니다.
이 섹션에서 나오는 용어 풀이
  • 오케스트레이션 — 여러 작업자와 도구가 서로 엉키지 않게 순서와 역할을 조율하는 일입니다. 지휘자가 오케스트라를 지휘하는 것과 비슷합니다.
  • 혼합 공급자 팀 — Codex, Claude, Gemini 같은 여러 AI 도구를 한 팀 안에서 함께 쓰는 방식입니다.
  • Codex / Claude / Gemini — 서로 다른 AI 도구 또는 AI 실행 환경입니다. OMX는 이들을 섞어 팀처럼 쓸 수 있습니다.
  • worktree — 하나의 Git 저장소에서 작업 공간을 여러 개로 나눈 것입니다. 여러 사람이 동시에 고쳐도 서로 파일을 덮어쓰지 않게 돕습니다.
  • Git — 파일 변경 이력을 기록하고 되돌릴 수 있게 하는 버전 관리 도구입니다.
  • API — 프로그램끼리 서로 기능을 호출하기 위해 정한 약속입니다.
  • JSON — 데이터를 중괄호와 키-값 형태로 적는 형식입니다. 사람이 읽고 프로그램도 읽기 쉽습니다.
  • 상태 루트 — 팀 상태 파일을 어디에 저장하고 읽을지 정하는 기준 폴더입니다.
  • 리더 — 팀 전체 방향을 정하고 작업자를 배정하며 최종 결과를 검토하는 중심 역할입니다.
  • 작업자 / worker — 팀 모드에서 실제로 분석, 수정, 검증 같은 일을 맡는 개별 AI 실행 단위입니다.
